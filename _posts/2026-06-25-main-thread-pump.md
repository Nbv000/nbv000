Unity — однопоточный рантайм. Большинство API вызывается только из main thread, и нарушение этого правила даёт либо тихую ошибку, либо крэш без стектрейса.

Когда работаешь через LD_PRELOAD или внешнюю библиотеку — у тебя есть свои потоки, но нет доступа к main thread напрямую. Нужен насос: механизм, который забирает задачи из очереди и выполняет их в нужном контексте.

## Подход

Цепляемся на метод, который Unity гарантированно вызывает из main thread на каждом кадре. Идеальный кандидат — `Update`-методы пользовательских компонентов. В Albion это `LocalInputHandler::Update` — легко находится в дампе, вызывается стабильно, не слишком нагружен логикой.

Inline-hook через detour: сохраняем оригинальный prologue, вставляем jmp на свой код, в своём коде дёргаем очередь задач, потом падаем в оригинал.

```cpp
// Очередь задач, доступная из любого потока
static std::mutex queue_mtx;
static std::queue<std::function<void()>> task_queue;

void enqueue(std::function<void()> fn) {
    std::lock_guard<std::mutex> lock(queue_mtx);
    task_queue.push(std::move(fn));
}

// Вызывается из main thread на каждом кадре
void pump() {
    std::lock_guard<std::mutex> lock(queue_mtx);
    while (!task_queue.empty()) {
        task_queue.front()();
        task_queue.pop();
    }
}
```

Детур на `LocalInputHandler::Update` — вставляем вызов `pump()` перед вызовом оригинала. Всё что попало в очередь через `enqueue()` выполнится в следующем кадре из main thread.

## Где это нужно

**Texture2D.Apply()** — применяет данные текстуры на GPU, только main thread. Если вызвать из фонового потока — тихо ничего не произойдёт или крэш в зависимости от версии Unity.

**GameObject.Find(), Object.FindObjectOfType()** — читают internal object table, которая модифицируется из main thread. Concurrent access без синхронизации даёт недетерминированный результат.

**Любой IL2CPP-метод, работающий с managed heap** — GC может запускаться в любой момент, и если managed объект аллоцируется не из main thread, это может привести к corruption в зависимости от настроек GC.

## Подводные камни

`LocalInputHandler::Update` может отсутствовать в дампе под другим именем или быть обфусцирован. В этом случае ищем любой другой `MonoBehaviour::Update` с подходящей частотой вызовов — метрики через счётчик вызовов в hook.

Размер prologue для detour зависит от архитектуры и конкретных инструкций на входе метода. На x86-64 минимум 5 байт для rel32 jmp, но если там уже есть короткий jmp или call — нужен длиннее. Используй disassembler для точного подсчёта, иначе сломаешь следующую инструкцию.

Если Unity собран с IL2CPP и включена компиляция AOT, адрес метода фиксирован относительно базы модуля. ASLR смещает весь модуль целиком, но relative offset между методами не меняется. Базу берём через `dlopen` / `GetModuleHandle`.

## Результат

Любой код, требующий main thread, упаковывается в лямбду и кладётся в очередь — выполняется в следующем кадре без каких-либо модификаций игрового кода. Накладные расходы на пустой кадр — несколько микросекунд на lock/unlock мьютекса.
