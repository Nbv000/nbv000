Воспроизводится стабильно: вызываешь `Texture2D.Reinitialize`, потом `GC.Collect()` или просто ждёшь следующего цикла сборщика — и получаешь SIGSEGV в нативном стеке где-то в `il2cpp::gc::GarbageCollector`.

Адрес падения — где-то вроде `0x0000000000000008` или `0x0000000000000018`. Классический null dereference со смещением.

## Что происходит

`Texture2D.Reinitialize` в определённых версиях Unity (проверено на 2020.x–2021.x) пересоздаёт нативный объект текстуры, но не всегда корректно обновляет managed wrapper. В результате `Il2CppObject*` для текстуры остаётся в managed heap, но его нативная часть уже указывает на освобождённую или переинициализированную память.

GC walker при следующем проходе видит живой managed объект, пытается пройти его поля по klass-descriptor, натыкается на невалидный pointer в нативной части — и падает.

Это не баг пользовательского кода. Это внутренняя несогласованность состояния между managed и нативным слоем, которая возникает именно после Reinitialize.

## Диагностика

Первый признак — SIGSEGV строго после Reinitialize, не в момент вызова, а через несколько кадров. Если крэш детерминированный по этому паттерну — причина здесь.

В `gdb` / `lldb` смотрим стек:

```
#0  il2cpp::vm::Class::GetFieldCount(Il2CppClass*)
#1  il2cpp::gc::GarbageCollector::RunFinalizers()
#2  il2cpp::gc::BoehmGC::GCCollect()
```

Или через Frida — hook на `GC.Collect` и логирование живых ссылок на текстуры перед вызовом. Если какой-то объект числится живым, но его `klass->fields` указывает в мусор — это он.

## Как лечится

**Вариант 1: не использовать Reinitialize.**

Вместо переинициализации создаём новую текстуру, копируем данные, удаляем старую. Дольше, но безопасно.

```csharp
// Вместо texture.Reinitialize(w, h)
var newTex = new Texture2D(w, h, texture.format, false);
// ... копируем пиксели ...
Object.Destroy(texture);
texture = newTex;
```

**Вариант 2: форс-финализация до GC.**

Если Reinitialize нужен — вызываем `GC.Collect()` вручную сразу после него, до того как Unity запустит свой цикл. Это даёт GC возможность обработать объект в контролируемый момент, а не во время произвольного кадра.

```csharp
texture.Reinitialize(w, h);
texture.Apply();
GC.Collect();
GC.WaitForPendingFinalizers();
```

Не устраняет причину, но делает крэш воспроизводимым в безопасный момент — там, где ты контролируешь состояние.

**Вариант 3: через LD_PRELOAD, если инъекция.**

Hook на il2cpp GC walker, добавляем проверку валидности klass-pointer перед разыменованием:

```cpp
// В своём detour перед вызовом оригинального walker
if (!is_valid_ptr(obj->klass) || !is_valid_ptr(obj->klass->fields)) {
    // пропускаем объект, не падаем
    return;
}
```

`is_valid_ptr` — проверка через `/proc/self/maps` или mincore(). Дорого, но этот путь только для диагностики и отладки, не для продакшна.

## Почему walker не защищён сам

Boehm GC (который использует Unity до определённых версий) предполагает, что managed heap консистентен. Он не проверяет валидность указателей при обходе — это обязанность runtime. Когда Reinitialize нарушает консистентность, GC идёт в segfault без какой-либо защиты.

В новых версиях Unity с incremental GC поведение может отличаться — там иная стратегия обхода. Но на legacy Boehm это воспроизводимо.

## Итог

Паттерн `Reinitialize → GC → SIGSEGV` — специфичный, но встречается в реальных проектах где текстуры обновляются динамически. Лечится либо заменой Reinitialize на create/destroy, либо форс-GC в нужный момент. Patch на walker — только для диагностики.
