Оба инструмента решают одну задачу: вклиниться в чужой процесс и изменить его поведение. Но работают по-разному, и выбор зависит от задачи.

## LD_PRELOAD

Переменная окружения, которая говорит линкеру загрузить твою библиотеку до всех остальных. Твои символы перекрывают символы из стандартных библиотек — классика для перехвата `malloc`, `open`, `connect`.

Преимущества:
- Никаких внешних зависимостей. Только твоя `.so` и сам процесс.
- Полный контроль над временем инициализации — конструктор библиотеки запускается до `main`.
- Перехват libc-функций тривиален: объявляешь функцию с тем же именем, внутри вызываешь оригинал через `dlsym(RTLD_NEXT, ...)`.
- Продакшн-режим: библиотека компилируется, деплоится, работает без внешних агентов.

Ограничения:
- Работает только если ты контролируешь запуск процесса. Для уже запущенного — нужен `ptrace` или другие механизмы.
- Не перехватывает прямые syscall (если код использует `syscall()` минуя libc).
- Требует перекомпиляции для каждого изменения. Итерация медленная.
- Anti-cheat и защиты проверяют `LD_PRELOAD` — тривиально детектируется.

## Frida

Динамический инструментационный фреймворк. Инжектирует движок V8 (JavaScript runtime) в целевой процесс через `ptrace`, после чего ты пишешь hooks на JavaScript, которые исполняются внутри процесса.

Преимущества:
- Аттач к живому процессу без перезапуска.
- Скриптовый язык — итерация быстрая. Поменял скрипт, перезапустил — нет перекомпиляции.
- `Interceptor.attach` работает по адресу или символу, не только по libc-экспортам.
- `Memory.scan`, `Memory.readByteArray` — удобный доступ к памяти из скрипта.
- Поддержка Android/iOS — там LD_PRELOAD недоступен или неудобен.

Ограничения:
- Внешний агент всегда присутствует в системе (`frida-server` или вшитый gadget). Детектируется.
- V8 внутри процесса — это потребление памяти и возможные конфликты с GC целевого процесса.
- Производительность: интерпретируемый JS для каждого вызова дорог. На высокочастотных методах — заметные просадки.
- Для продакшна не годится — это инструмент исследования.

## Критерии выбора

**Frida, если:**
- Нужно быстро изучить поведение процесса, найти нужный метод, поиграть с аргументами.
- Процесс уже запущен, перезапуск невозможен или нежелателен.
- Работа на Android/iOS.
- Одноразовый эксперимент, не требующий production-кода.

**LD_PRELOAD, если:**
- Нужно решение, которое работает в продакшне без внешних агентов.
- Перехватываешь libc-функции (`malloc`, `read`, `connect`, `dlopen`).
- Нужна максимальная производительность — нативный код без JS-прослойки.
- Хочешь контролировать инициализацию до старта основного кода.

**Frida для разведки → LD_PRELOAD для внедрения** — рабочий паттерн. Frida находит нужный адрес и проверяет гипотезу, LD_PRELOAD реализует решение.

## Пример: перехват сетевого соединения

Frida (разведка):
```javascript
Interceptor.attach(Module.findExportByName(null, "connect"), {
    onEnter(args) {
        // смотрим куда коннектится процесс
        console.log(JSON.stringify(Socket.peerAddress(args[0])));
    }
});
```

LD_PRELOAD (решение):
```cpp
int connect(int sockfd, const struct sockaddr* addr, socklen_t addrlen) {
    // подменяем адрес назначения или логируем
    auto real_connect = (decltype(&connect))dlsym(RTLD_NEXT, "connect");
    return real_connect(sockfd, &patched_addr, addrlen);
}
```

## Inline hook как третий вариант

Когда нужно перехватить не libc-функцию, а произвольный адрес внутри бинаря — ни LD_PRELOAD, ни Frida не дают удобного нативного решения в продакшне. Inline hook (detour patching) через свою `.so`, загруженную LD_PRELOAD — это сочетание обоих подходов. Загрузка через LD_PRELOAD, перехват через прямой патч кода.
